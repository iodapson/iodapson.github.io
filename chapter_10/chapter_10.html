Managing Growing Projects with Packages, Crates, and Modules

1. What are the four instruments of code organization that Rust provides

  • Packages
  • Crates
  • Modules (and use)
  • Paths

2. What is a Rust package?

It is a Cargo feature that lets you build, test, and share crates. A package contains a 'Cargo.toml' file that decribes how to build those crates.

3. What are Rust crates?

They are a tree of modules that produce a library or executable.

4. What are Rust modules (and use)?

They let you control the organization, scope and privacy of paths.

5. What are Rust paths?

Paths are a way of naming an item, such as a struct, function, or module.

6. What is a crate root?

It is a source file that the Rust compiler starts from and makes up the root module of your crate. A crate is a binary or library.

7. A package can contain at most how many library crates?

One.

8. What must a package contain at the very least?

One crate (either binary or library crate).

9. Where are binary crates stored inside a package?

Inside 'src/bin'. This means that a package can contain many binary crates, all of which would be stores inside the 'src/bin' directory.

10. What file constitutes the binary crate root to Rust?

'src/main.rs'.

11. What file constitutes the library crate root to Rust?

'src/lib.rs'.

12. What does Rust's Cargo do with crate root files?

Cargo passes the crate root files to "rustc' to build the libray or binary, depending on what crate root it finds.

13. How do you point to say trait 'Rng' inside Rust's "rand" crate?

With syntax: rand::Rng

14. Mention two functions/advantages that modules bring to code base structuring and organization

  • Modules let us organize code within a crate into groups for readability and easy reuse.
  • Modules control the privacy of items, which is whether an item can be used by outside code (ie, is public), or is an internal implementation detail and not available for outside (ie, is private).

16. The crate root files src/main.rs and src/lib.rs implicity form a top-most module in the crate's module structure. What is the name of the implicitly formed top-most module?

"crate", as in.. crate

17. Specifying the path to a an item you want to bring in (import/use) inside your code can be done in two path ways, what are they?

  • an absolute path; starts from a crate root by using a crate name or literal 'crate'.
  • a relative path; starts from the current module and uses 'self', 'super', or an identifier in the current module.

18. Since Rust code in general (especially code inside modules) are private by default, what keyword do you use to make them public, such that they can be brought in / used inside your code?

The "pub" keyword, as in; pub fn add_to_waitlist() {}

19. Demonstrate code that creates one or more modules and use code inside the module by means of an absolute path

Here:

  mod front_of_house {
    pub mod hosting {
      pub fn add_to_waitlist() {}
    }
  }
  // now we'll use the function add_to_waitlist()
  pub fn eat_at_restaurant() {
    // the absolute path specification
    crate::front_of_house::hosting::add_to_waitlist();
  }

20. Demonstrate code that creates one or modules and use code inside the module by means of a relative path

Here:

  mod front_of_house {
    pub mod hosting {
      pub fn add_to_waitlist() {}
    }
  }
  pub fn eat_at_restaurant() {
    // Relative path
    front_of_house::hosting::add_to_waitlist();
  }

The "super" keyword, as in.. super

23. Describe the behavior of Rust's struct fields behavior, in restropective of being public or private

All struct fields are private by default even if the struct itself has been made public. To make a struct's field public, you have to explicitly do so via the "pub" keyword.

24. Describe the behavior of Rust's enum variants, in restrospective of being public or private

All enum variants are public by default, provided their enum has been made public, so you don't have to utilize the "pub" keyword to make an enum's variant public. All you need do is simply make the enum itself public via the "pub" keyword.

25. What is the implication of Rust's struct fields behavior, in restropective of being public or private?

It means that you have to cater for private struct fields by providing an implementation of the struct, using "impl" Struct_Name to create a method/associated function which then returns the struct instance with the private field(s) assigned respective values. If you don't do this, you would end up with code that does not compile when you try to call any of the public fields inside a struct with an unimplemented private field. You still won't be able to access the implemented private field from outside code though.

26. Demonstrate code that implements a struct with a private field to provide a method/associated function which returns an instance of the struct itself which then implements the private field

Here:

  mod back_of_house {
    pub struct Breakfast {
      pub toast: String,
      seasonal_fruit: String, // a private field
    }

    impl Breakfast {
      pub fn summer(toast: &str) -> Breakfast {
        Breakfast {
          toast: String::from(toast),
          seasonal_fruit: String::from("peaches"),
        }
      }
    }

  }

  pub fn eat_at_restaurant() {

    // Order a breakfast in the summer with Rye toast
    let mut meal = back_of_house::Breakfast::summer("Rye");

    // Change our mind about what bread we'd like
    meal.toast = String::from("Wheat");

    println!("I'd like {} toast please", meal.toast); // The next line won't compile if we uncomment it; we're not allowed
    // to see or modify the seasonal fruit that comes with the meal
    // meal.seasonal_fruit = String::from("blueberries");
  }

Note that you access an enum or struct impl method using syntax "enum_variant.method_name" or "struct_instance.method_name", respectively. Whereas, you access a struct's impl associated function using syntax "struct_instance::method_name()". All struct fields are accessed with syntax "struct_instance.field_name".

  mod front_of_house {

30. What do you do when you need to use two or more (external) code items who happen to have the same name?

Apply the use..as construct so you can introduce one or more of the external code item's path(s) into your code under a different name alias

32. What construct do you use to re-export a code item brought into scope, ie use, so that it becomes available for others (other code items) to bring them into their scope

The "pub use" construct As in; pub use

33. Demonstrate code that utilizes the pub..use construct to expose code items used in your own code to other code

Here:

  mod front_of_house {
    pub mod hosting {
      pub fn add_to_waitlist() {}
    }
  }
  pub use crate::front_of_house::hosting;
  pub fn eat_at_restaurant() {
    hosting::add_to_waitlist();
    hosting::add_to_waitlist();
    hosting::add_to_waitlist();
  }

N.B: Re-exporting is useful when the internal structure of your code is different from how programmers calling your code would think about the domain. For example, in this restaurant metaphor, the people running the restaurant think about "front of house" and "back of house". But customers visiting a restaurant probably won't think about the parts of the restaurant in those terms. With pub..use, we can write our code with one structure but expose a different structure. Doing so makes our library well organized for programmers working on the library and programmers calling the library.

34. Demonstrate code that use nested paths "std::cmp::Ordering" and "std::io"

  // --snip--

35. Demonstrate code that use a nested path alongside another path represented with the "self"

Here:

  use std::io::{self, Write};

The above code is equivalent to the following:

  use std::io;
  use std::io::Write;

36. What is the glob operator in Rust represented by?

A "_", as in; _

37. What is the glob operator in Rust used for?

It is construct that allows you to bring all public items defined in a path into scope. You can specify the path followed by "*", the glob operator.

38. Demonstrate code that utilizes the glob operator

Here: use std::collections::*; This 'use' statement above brings all public items defined in "std::collections" into the current scope. Be careful when using the glob operator! Glob can make it harder to tell what names are in scope and where a name used in your program was defined.

39. Mention some common use-cases of the glob operator

  • The glob operator is often used when testing to bring everything under test into the "tests" module.
  • The glob operator is also sometimes used as part of the prelude pattern. See https://doc.rust-lang.org/std/prelude/index.html#other-preludes ..for more information on that pattern

40. What do you when you need to bring code for use from large modules defined across separate files into your code?

By using keyword 'mod' and the name of the file you wish to consume, for example, like this: mod name_of_rust_file

41. Demonstrate code that brings in code from one module into another source code / crate, be it a library crate (lib.rs), or a binary crate (main.rs)

Imagine you have a Rust source file inside directory 'src/front_of_house.rs', with content:

  pub mod hosting {
    pub fn add_to_waitlist() {}
  }

..whose code you would like to use in a different source code, like lib.rs or main.rs. ..And you have an actual library code (lib.rs) located in file directory src/lib.rs, which would consume 'src/front_of_house.rs' shown earlier ..where the consumer lib.rs code looks like this:

  mod front_of_house;
  pub use crate::front_of_house::hosting;
  pub fn eat_at_restauratant() {
    hosting::add_to_waitlist();
    hosting::add_to_waitlist();
    hosting::add_to_waitlist();
  }

** ...the line 'mod front_of_house;' implicitly links the code inside front_of_house.rs into the consumer code of lib.rs. That is how you consume code (modules) accross multiple files.

Summary:

Rust lets you split a package into multiple crates and a crate into modules so you can refer to items defined in one module from another module. You can do this by specifying absolute or relative paths. These paths can be brought into scope with a 'use' statement so you can use a shorter path for multiple uses of the intem in that scope. Module code is private by default, but you can make definitions public by addding the pub keyword.
