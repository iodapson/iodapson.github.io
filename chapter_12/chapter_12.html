Error Handling

1. What are the two major categories of errors which Rust provides?

  • recoverable errors
  • unrecoverable errors

2. How do you handle recoverable errors?

By making use of the Result<T, E> generic enum

3. How do you specify unrecoverable errors?

By making use of the 'panic!()' macro. You simply call it.

4. What does Rust do by default when it sees an explicity panic! call or an implicit panic code occurs (such as acessing an array at an index that exceeds the array length)?

It causes your program to start to unwind, which basically means Rust walks back up the stack and cleans up the data from each function it encounters.

5. What is the alternative to unwinding a program, which takes a lot of work?

abort

6. How do you change the default behavior of your program from unwinding a panic code program to simply aborting the panic code program?

By making a specific change/addition in your Cargo.toml file, as follows;

[profile]
panic = 'abort'

7. How do you change the default behavior of your program in release mode, such that it aborts upon encountering a panic code, instead of unwinding

By making the following specific change/addition to your project's Cargo.toml file, as follows;

[profile.release]
panic = 'abort'

Here:

  fn main() {

  }

9. Demonstrate code which causes an implicit call to the panic!() macro

Here:

  fn main() {
    let v = vec![1, 2, 3];
    v[99]; // this line causes your program to panic
  }

10. What command do you use to run or build a Rust program which panics, and unwinds, in order debug and see where the error transpired.

$ RUST_BACKTRACE=1 cargo run

..or, in the case that you simply just want to build your project

$ RUST_BACKTRACE=1 cargo build

11. Demonstrate code that recovers an error by means of using the Result<T, E> enum instead of letting the code automatically implicitly panic

Here:

  use std::fs::File;
  fn main() {
    let f = File::open("hello.txt");
    let f = match f {
      Ok(file) => file,
      Err(error) => {
        panic!("There was a problem opening the file: {:?}", error);
      },
    };
  }

12. Is Result<T, E> part of Rust's prelude?

Yes.

13. Demonstrate code which recovers from an error using Result<T, E> of course, and then matches against the kind of error E, in order to yet again recover from it. That's two levels of error recovery

Here:

  use std::fs::File;
  use std::io::ErrorKind;
  fn main() {
    let f = File::open("hello.txt");
    let f = match f {
      Ok(file) => file,
      Err(error) => match error.kind() {
        ErrorKind::NotFound => match File::create("hello.txt") {
          Ok(fc) => fc,
          Err(e) => panic!("Problem creating the file: {:?}", e),
        },
        other_error => {
          panic!("Problem opening the file: {:?}", other_error)
        }
      },
    };
  }

14. Demonstrate code that recovers from an error without using a Result<T, E>, but instead uses the 'unwrap_or_else()' closure syntax

Here:

  use std::fs::File;
  use std::io::ErrorKind;
  fn main() {
    let f = File::open("hello.txt").unwrap_or_else(|error| {
      if error.kind() == ErrorKind::NotFound {
        File::create("hello.txt").unwrap_or_else(|error| {
          panic!("problem creating the file: {:?}", error);
        })
      } else {
        panic!("Problem opening the file: {:?}", error);
    });
  }

15. How does the 'unwrap()' work?

It works on what it is called on which returns a Result<T, E>, such that it either returns T in the Result<T, E>,, or panics if the Result<T, E> is an error value.

16. Demonstrate code which uses the unwrap() syntax to recover an error or panic

Here:

  use std::fs::File;
  fn main() {
    let f = File::open("hello.txt").unwrap();
  }

17. What is the difference between syntax 'unwrap()' and 'expect()'?

'expect()' lets you pass in an error message in case the code it is called on returns an error, whereas, 'unwrap()' does not give you such opportunity.

18. Demonstrate 'expect()' code in action

Here:

  use std::fs::File;
  fn main() {
    let f = File::open("hello.txt").expect("Failed to open hello.txt");
  }

19. Demonstrate function code that is error sensitve and redundant, i.e, handles erros

Here:

  use std::fs::File;
  use std::io::{self, Read};

  // the function that is error redundant
  fn read_username_from_file() -> Result<String, io::Error> {
    let f = File::open("hello.txt");
    let mut f = match f {
      Ok(file) => file,
      Err(e) => return Err(e),
    };
    let mut s = String::new();
    match f.read_to_string(&mut s) { // read the contents of file f into String s
      Ok(_) => Ok(s),
      Err(e) => Err(e),
    }
  }

20. What is the shortcut operator for propagating errors in Rust,?

The '?' operator

21. Demonstrate code that uses the '?' operator to propagate errror, such that the code is equivalent to the code in no-19, above

Here:

  use std:fs::File;
  use std::io;
  use std::io::Read;
  fn read_username_from_file() -> Result<String, io::Error> {
    let mut f = File::open("hello.txt")?;
    let mut s = String::new();
    f.read_to_string(&mut s)?;
    Ok(s)
  }

You could even chain method calls, using the '?' operator, e.g, like this;

  use std::fs::File;
  use std::io;
  use std::io::Read;
  fn read_username_from_file() -> Result<String, io::Error> {
    let mut s = String::new();
    File::open("hello.txt")?.read_to_string(&mut s);
    Ok(s)
  }

22. Can you use the '?' operator inside a function that does not return either Result<T, E> or Option?

No, so the following code for example would result to an error

  // ERRORNOUS CODE
  use std::fs::File;
  fn main() {
    let f = File::open("hello.txt")?;
  }

23. Can you mix-match Result and Option in a function that uses the ? operator?

No.

24. Demonstrate code that uses the '?' error handling operator in function that returns Option

Here:

  fn last_char_of_first_line(text: &str) -> Option<char> {
    text.lines().next()?.chars().last()
  }
