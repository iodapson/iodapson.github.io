References and Borrowing, The Slice data type

1. What is a reference?

A reference is like a pointer that can be used to access data stored at a given address.

2. How is a reference different from a pointer?

Unlike a pointer, a reference is guaranteed to point to the valid actual value of a particular type.

3. Demonstrate code that creates a function that has a parameter (argument) which is reference to an object (as in, a reference based type) without taking ownership of the argument value passed in to the function

Here:

fn main() {
  let s1 = String::from("hello");
  let len = calculate_length(&s1);
  println!("The length of '{}' is {}.", s1, len);
}
fn calculate_length(s: &String) -> usize { // s is a reference to a String
  s.len()
} // Here, s goes out of scope. But because it does not have ownership of what it refers to, hence, the dropping off of its source (s1) does not happen.

6. What happens when you try to modify something, for example, a variable that you're borrowing?

Such action would result to a compile error.

7. Are references immutable by default?

Yes.

8. Demonstrate code that makes it possible to modify something you're borrowing

Here:

fn main() {
  let mut s = String::from("hello"); // first make the target source variable that will be borrowed mutable
  change(&mut s); // pass in a mutable reference argument to the function (change) which will mutate the borrowed variable
}
fn change(some_string: &mut String) {
  some_string.push_str(", world");
}

9. What is the single biggest restriction of mutable references?

You can have only one mutable reference to a particular piece of data at a time. For example the following code piece would not work:

fn main() {
  let mut s = String::from("hello");
  let r1 = &mut s; // first mutable reference to s
  let r2 = &mut s; // second mutable reference to s. Illegal op.
  println!("{}, {}", r1, r2);
}

10. Why is there a limitation on the number of times you can mutate a reference?

To allow mutation but in a very controlled fashion.

11. Mention three behaviors that can result to race condition or data race

  • When two or more pointers access the same data at the same time.
  • At least one of the pointers to a data is being used to write to the data.
  • When there's no mechanism being used to synchronize access to the data.

12. Demonstrate code that creates more than one mutable reference, one at a time, by way of using braces for scoping

Here:

  let mut s = String::from("hello");
  {
    let r1 = &mut s;
  } // r1 goes out of scope here, so we can make a new reference with no problems.
  let r2 = &mut s;

13. Can you combine an immutable reference with a mutable variable?

Yes and No. Yes, with the exception that you use only one at a time. That is, you can use the immutable one(s) first, and then use the mutable one, without using the immutable again, ever, as this would create a big problem.

14. Can you create two or more subsequent immutable reference?

Yes

15. Demonstrate code that legally combines an immutable and a mutable reference, and illustrating the proper way to combine a mutable and immutable reference.

Here:

let mut s = String::from("hello");
let r1 = &s; // no problem
let r2 = &s; // no problem
println!("{} and {}", r1, r2);
// variables r1 and r2 will not be used again after this point
let r3 = &mut s; // no problem
println!("{}", r3);

16. What is the ability of the compiler to tell that a reference is no longer being used at a point before the end of the scope called?

Non-Lexical Lifetimes (NLL for short).

17. Does Rust leave room for the possiblity of dangling references, e.g, looking at the following piece of code

  fn main() {
    let reference_to_nothing = dangle();
  }
  fn dangle() -> &String {
    let s = String::from("hello");
    &s
  } // note, 's' goes out of scope and is deallocated here, hence &s returned by dangle() is a dangling reference.

No. Rust won't let us do this. The solution to the above errornous code in the question is to return the data itself that s holds as in:

fn no_dangle() -> String {
  let s = String::from("hello");
  s
}

Although String is a reference type being returned, it is still different from an explicit pointer/"reference" denoted with an '&', hence the code works.

18. Describe Rust's "Slice"Type (Slices)

Slices let you reference a contiguous sequence of elements in a collection rather than the whole collection.

19. Does a slice have ownership?

No.

20. Explain why a slice does not have ownership

Becuase a slice is a kind of reference, it does not have ownership.

21. Explain the code snippet

let s = String::from("something");
let bytes = s.as_bytes();

The "as_bytes()" method returns the string it is called on as an array of bytes.

26. What is the equivalent code to this code;

Answer: let s = String::from("hello"); let len = s.len(); let slice = &s[3..];

29. Slice range indices must occur at valid UTF-8 character boundaries, what does this mean?

It means that if you attempt to create a sling slice in the middle of a multibyte character, your program will exit with an error. You can read more thorough discussion of UTF-8 handling in the "Storing UTF-8 Encoded Test with Strings" at; https://doc.rust-lang.org/book/ch08-02-strings.html#storing-utf-8-encoded-text-with-strings

30. Demonstrate code that returns the first word in a String. Hint;

The first word can be detected by by sniffing out the fist white space that occurs in the String. Letters prior to the spacing will constitute a word.

Here:

  fn first_word(s: &String) -> &str {
    let byte = s.as_bytes();
    for (i, &item) in bytes.iter().enumerate() {
      if item == b' ' {
        return &s[..i];
      }
    }
    &s[..]
  }
