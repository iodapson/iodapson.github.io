More about Cargo and crates.io

1. What are release profiles in Rust?

They are predefined and customizable profiles with different configurations that allow a programmer to have control over various options for compiling code. Each profile is configured independently of the others.

2. What are the two main profiles that Cargo has?

  • build (when you run 'cargo build'), which has good defaults for development.
  • release (when you run 'cargo build --release') which has good defaults for release builds.

3. How can you override any subset of the default settings of a profile?

By adding a [profile.*] section in your Rust project's Cargo.toml file.

4. Showcase an example where you override a default setting in both dev and release profiles

Filename: Cargo.toml

[profile.dev]
opt-level = 0

[profile.release]
opt-level = 3

5. What is a documentation comment?

They are comments that will generate an HTML display of documentation for public API items in your code so programmers can know how to use your crate as opposed to how your crate is implemented.

6. How do you write a documentation comment in Rust?

By using three slashes ///. You create headers with the # character, and you annotate demo code with an opening and closing pair of triple ticks (```)

7. Create a sample documentation comment

Here:

  /// Adds one to the number given.
  ///
  /// # Examples
  ///
  /// ```
  /// let arg = 5;
  /// let answer = my_crate::add_one(arg);
  ///
  /// assert_eq!(6, answer);
  /// ```
  pub fn add_one(x: i32) -> i32 {
    x + 1
  }

8. What Cargo command do you use to actually generate HTML documentation from a documentation comment?

$ cargo doc

9. What Cargo command option do you use to generate and open HTML documentation from a documentation command?

$ cargo doc --open

  • Panics. Discus scenerios in which the function being documented could panic
  • Errors: If the function returns Result, describe the kind of errors that may occur, what could provoke such error(s), and possible optimal ways to handle them.
  • Safety - unsafe functions could have an explaination why they are unsafe, and the invariants of their call stated.

11. What happens to demo code inside a documentation comment when you run Cargo command 'cargo test'?

It would run the demo code as a test as well.

12. What documentation comment syntax do you use to specify documentation for only a specify item that follows the last occurence of it?

'//!', as in: //! This means that an item that occurs after the last occurrence of //! gets documented.

13. What kind of documentation is especially suited for documenting a crate as a whole, or a module?

15. Why would you apply 'pub use' to your code to help developers understand how to use your library code, for example?

So you can re-export pieces of your code to the public to see at first glance when they run 'cargo doc --open'. This would utimately show them how to use your code since your code's structure is exposed to them at first glance. In addition, they now have access to deeply nested pieces right from top-level modules.

17. What steps are to be followed to create a crates.io account, login to it, and then ultimately publish a project to crates.io

  • Create an account at crates.io using a GitHub account
  • Retrieve your craite.io API key from https://crates.io/me/
  • Login to your crate.io account using the Cargo command; $ cargo login <your_api_key_goes_here>
  • Add the required metadata to Cargo.toml under [package]
  • Run Cargo command 'cargo publish'

N.B: As with other API keys, you should also keep your crates.io API key private. You also have to make sure your project name under [package] in your Cargo.toml file is unique in crates.io.

18. What are some common metadata you should include inside your project's Cargo.toml file under its [package] section before publishing it to crates.io?

  • name
  • version
  • edition
  • description
  • license Note that if your preferred license is not included in the Software Package Data Exchange (SPDX) [http://spdx.org/licenses/] list, then you must include the license content as a file inside your project's top-level folder, and then have using property 'license-pair' instead of 'license' to specify the license file name, as its value.

19. List example meta-data for a project required for a sample guessing_game Rust project to be published on crates.io

Here:

description = "A fun game where you guess what number the computer has chosen."
license = "MIT OR Apache-2.0"

N.B: Notice how OR was used to specify more than one license for the project.

20. What do you do to publish a new version of your existing published crate?

Simply change the 'version' property value specified in your Cargo.toml file and republish (using Cargo command 'cargo publish'), using the Semantic Versioning rules.

21. What do you do to prevent someone's future project from adding a specific version of your crate as a new dependency?

By using the 'cargo yank' command. For example, like this; $ cargo yank --vers 1.0.1 The above command yanks version 1.0.1 of the project, thus preventing other projects from depending on it moving forward.

22. What do you undo a yanked version of a Rust project?

$ cargo yank --vers 1.0.1 --undo

N.B: A yank does not delete any code. For exampl, the yank feature is not intended for deleting accidentally uploaded secrets. If that happens, you must reset those secrets immediately.

23. What is a workspace?

It is a set of packages that share the same 'Cargo.lock' and output directory.

24. Why would you want to use workspaces?

Because as your Rust project develops, you might find that the library crate of your project continues to get bigger and you want to split up your package further into multiple library crates.

25. What are the steps to follow in creating a Rust workspace

  • First, create a directory that would serve as the workspace directory

    $ mkdir workspace_dir_name
$ cd workspace_dir_name

  • Second, create a 'Cargo.toml' file inside the 'workspace_dir_name' directory. Note that this time, the 'Cargo.toml' will have a [workspace] section, but not have a [package] section

  • Third, add the sub-cargo project names (one binary crate, and as many library crates as required) that will be contained inside the 'workspace_dir_name', like so;

    [workspace]
members = [
  "member_project_one",
  "member_project_two",
]

  • Fourth, create the sub-cargo projects whose names you added as members inside the workspace_dir_name's Cargo.toml file. You simply create a normal Cargo project using Cargo command;

    $ cargo new member_project_one

to create a binary crate. Ideally, should be just one.

OR

$ cargo new memeber_lib_project --lib

..run from the top level folder inside your workspace_dir_name.

  • Fifth, run Cargo command;
$ cargo build

26. What file structure would be created when you run Cargo command 'cargo build' for the very first time for your Cargo workspace project?

|-Cargo.lock
|-Cargo.toml
|-member_project_one
  |-Cargo.toml
  |-src
    |-main.rs
|-member_project_two
  |-Cargo.toml
  |-src
    |-lib.rs
|-target

Note that it would not matter where you run 'cargo build' so long as it is either within the top-level folder or a member_project inside your workspace_dir_name directory.

27. What's the relevance of the 'target' folder inside a workspace's top level folder?

It is where compiled artifacts for your workspace code is placed into.

28. How do you make a binary crate of a sub-project (member_project_one) inside a workspace, utilize code from another sub-project crate (member_project_two, a lib crate) within the same workspace?

By updating the Cargo.toml file of the binary crate's sub-project folder to look similar to the following;

  [dependencies]
  member_project_two = {path = "../member_project_two"}

..and then add; use member_project_two;

..to the member_project_one binary crate's main.rs file.

29. What happens when you have a dependency, for example, 'rand' which you specify in in each member sub-projects (crates) of a workspace. How does Cargo handle member projects having multiple dependencies of the same library crate 'rand' from crates.io?

Cargo will resolve the versions of rand across member crates of the workspace into one version, and then record that resolved one version of 'rand' in the workspace's Cargo.lock file. N.B: Even though a dependency might be recorded in a workspace's Cargo.lock file, you still need to explicitly add such dependency into a member crate's cargo.toml file to actually be able to use the dependency in the member crate.

30. Do member crates of a workspace each have their own Cargo.lock file?

No. They only have their own respective Cargo.toml file. Member crates of a workspace get to share only one Cargo.lock file, which will be located at the top-level folder of the workspace directory itself.

31. What Cargo command do you use to single out a member crate in a workspace to be run by Cargo?

$ cargo run -p name_of_the_member_crate

32. What Cargo command do you use to single out a member crate in a workspace to be tested by Cargo?

$ cargo test -p name_of_the_member_crate

33. When you wish to publish member crates in a workspace, is it possible to publish the member crates at once?

No since 'cargo publish' does not support '--all' or '-p' flag.

34. How do you publish member crates in a workspace?

By publishing them separately. You'll run 'cargo publish' invidividually from each member crate inside the workspace.

N.B: As your project grows, consider using a workspace: it's easier to understand smaller, individual components than one big blob of code. Furthermore, keeping the crates in the a workspace can make coordination between them easier if they are often changed at the same time.

35. You can install packages / Cargo projects that don't have binary targets, true or false?

False. You can only install packages that have binary targets. N.B: A binary target is a runnable program that is create if the crate/project has a 'src/main.rs' file, or another file specified as a binary, as opposed to a library target that isn't runnable on its own but is suitable for including within other programs. Installing binary crates provide a convenient way for Rust developers to install packages that others have shared on crates.io as tools.

37. Where are all binaries installed with 'cargo install' stored?

In the installation root's 'bin' folder. If you installed Rust using rustup.rs and don't have any custom configurations, this directory will be $HOME/.cargo/bin. Ensure that directory '$HOME/.cargo/bin' is in your $PATH to be able to run programs you've installed with 'cargo install'. After installing a crate, e.g, 'installed_example_crate', and the installation directory of 'installed_example_crate' is in your $PATH, you can now use the crate by calling its name directly. Here's a demo: $ cargo install installed_example_crate` `$ installed_example_crate --help
